## 翻译 context 策略调研记录（2026-07-18）

起因：现有实现里，每一句译文的 `context` 字段 = 同一个事件页面/列表里除它自己以外的
**全部**其它台词（见 `mv_mz.py` / `_rgss_common.py` 的 `_pending_to_units`，
`wolf.py` 的 `_texts_with_sibling_context`），页面越长这部分开销越是平方级增长
（N 句台词 → 每句都带 N-1 句 context，DEFAULT 策略的批量 prompt 里还是每条各自
重复拼贴一份，不是共享一份）。怀疑这是翻译变慢的主因，于是拿上面那台机器的
`sakura-galtransl:latest` 做了组对照测试（脚本已经不在仓库里，是临时脚本，测试
方法：同一段 12 句对话场景，分别用"整段一次性提交""单条+全量 history""单条+
前后各 2 句 history""单条+零 history"四种方式喂给模型，比较 token 开销和译文
质量）。

**结论**：

1. **人称代词解析对 context 量不敏感**——测试里第 11 句「彼」指代第 4~6 句才出现
   的角色，无论给不给这段 history，模型都稳定翻成"他"，没有出现指代错误。日译中
   "彼"→"他"这种转换本身对上下文依赖不强，不能靠这类 case 证明"context 越多翻
   译越准"。
2. **真正的一致性问题是角色名音译在多次调用之间会漂移，而不是单次调用内的上下文
   够不够**——同一个角色名「ユウキ」在五次独立请求里被翻成了五个不同的名字（优无记/
   祐树/祐树/侑树/勇希），但在同一次请求内部（比如一次性提交 12 句的那次）名字是
   自洽一致的。换句话说：只要一段连续台词在**同一次 API 调用**里翻完，模型自己就能
   保持人名/称呼一致；一旦被拆成多次独立调用（不管每次带不带 context），一致性就
   会在调用之间跑掉——这不是靠塞更多 context 能解决的，得靠术语表/记忆库跨调用
   固定译名。
3. **token 开销上，"整页一次性提交"比"拆成多条单独请求（哪怕每条只带一点点
   context）"便宜得多**——12 句整段一次提交只花了约 322 prompt tokens；拆成 12 次
   单条请求（每条带前后 2 句 history）每次约 190 prompt tokens，12 次合计约
   2300+，接近 7 倍。差距主要来自每次请求都要重复付一遍固定模板/system prompt 的
   token，不是 context 内容本身占大头。

**方向性建议**（还没有实现，等和用户对齐后再动）：与其在现有"每句单独请求 + 各自
带一份 context"的结构里调小 context 窗口（±1/±2 句），不如把同一事件页面里连续
的台词按顺序打包成一个请求整体提交翻译（真正的"段落输入、段落输出"），这样：
上下文自然靠"同一次请求里的其它行"提供，不需要额外拼贴 context 字段；token
开销从"每条都要重复付模板税"降下来；人名一致性在段落内部自动保持。跨页面/跨请求
的译名漂移问题需要另外解决（术语表/`store.py` 的记忆库按角色名做强制统一），不是
这次 context 策略调整能顺带解决的。

DeepSeek/SiliconFlow 云端模型那侧的同类对照测试还没跑完（临时脚本执行时环境报了
